# MVP PRD
Ontology 기반 청원 + 후원 플랫폼 MVP 요구사항 정의서

## 1. 문서 개요

### 1.1 목적
이 문서는 청원 + 후원 + 자동분류 + 대시보드 기능을 가진 시민참여 플랫폼의 MVP 범위와 요구사항을 정의한다.

### 1.2 제품 비전
이 서비스는 사용자가 사회적 이슈에 대해 청원을 만들고, 다른 사용자가 서명과 후원을 통해 참여할 수 있게 하는 플랫폼이다.  
핵심 차별점은 다음 두 가지다.

1. 청원 등록 시 온톨로지 기반 자동분류
2. 서명/후원/인구통계 기반 운영 대시보드

### 1.3 MVP 목표
MVP의 목표는 다음을 검증하는 것이다.

- 사용자가 실제로 청원을 생성하고 탐색하고 서명할 수 있는가
- 자동분류가 운영 가능한 수준으로 청원을 대분류/중분류/소분류에 매핑할 수 있는가
- 서명자 통계를 집계해 운영자에게 의미 있는 대시보드를 제공할 수 있는가
- 후원 구조를 붙일 수 있는 도메인 모델이 안정적으로 설계되는가

---

## 2. 제품 범위

### 2.1 MVP 포함 범위
MVP에서 반드시 포함할 기능은 다음과 같다.

- 회원가입 / 로그인
- 청원 등록
- 청원 목록 / 상세 조회
- 서명 기능
- 카테고리 기반 탐색
- 자동분류 MVP
- 검토 큐(Review Queue)
- 청원별 기본 통계
- 운영자 대시보드
- 후원 구조의 기본 도메인 및 mock 흐름

### 2.2 MVP 제외 범위
다음은 MVP 이후 단계에서 다룬다.

- 실제 PG 결제 연동 완성
- 추천 시스템
- 유사 청원 자동 추천
- 고급 검색엔진(OpenSearch) 고도화
- 알림/이메일 자동화
- 다국어 지원
- 모바일 앱
- 청원 홍보 광고 시스템
- 고급 moderation automation
- AI 청원 문구 생성기

## 3. 핵심 사용자

### 3.1 일반 사용자
청원을 탐색하고, 서명하고, 필요시 후원하는 사용자

### 3.2 청원 작성자
직접 청원을 생성하고, 업데이트를 올리고, 참여 현황을 확인하는 사용자

### 3.3 운영자
청원 검토, 자동분류 검토, 통계 확인, 상태 변경을 수행하는 사용자

### 3.4 관리자
운영자 권한 관리, 시스템 설정, 대시보드 확인, 로그 확인을 수행하는 사용자

---

## 4. 문제 정의

기존 청원 플랫폼은 다음 문제가 있다.

- 청원 수가 많아질수록 운영자가 수동으로 분류하기 어렵다
- 다양한 사회 이슈를 일관된 구조로 분류하기 어렵다
- 서명자 데이터를 전략적으로 해석하기 어려운 경우가 많다
- 후원과 청원이 분리되어 사용자의 참여 흐름이 끊긴다

이 제품은 다음 문제를 해결한다.

- 청원 입력 시 자동분류로 운영 부담 감소
- 대/중/소 분류 체계로 탐색성 향상
- 서명자 통계 대시보드 제공
- 청원과 후원을 하나의 흐름으로 연결

---

## 5. 사용자 시나리오

### 5.1 청원 작성 시나리오
1. 사용자가 로그인한다
2. 청원 등록 화면에 진입한다
3. 제목, 본문, 대상기관, 지역, 해시태그를 입력한다
4. 시스템이 자동분류를 수행한다
5. confidence가 높으면 review 상태로 저장되고 운영 검토를 대기한다
6. 운영자가 확인 후 published 상태로 공개한다

### 5.2 서명 참여 시나리오
1. 사용자가 청원 상세 페이지에 진입한다
2. 청원 내용을 읽는다
3. 이름 또는 표시명, 지역, 연령대, 성별을 입력한다
4. 서명한다
5. 서명 수와 통계가 갱신된다

### 5.3 후원 참여 시나리오
1. 사용자가 청원 상세 페이지의 후원 영역에 진입한다
2. 일시후원 또는 정기후원을 선택한다
3. 금액을 선택한다
4. MVP에서는 mock 후원 성공 흐름으로 저장된다
5. 후원 통계에 반영된다

### 5.4 운영자 검토 시나리오
1. 운영자가 review queue를 확인한다
2. 자동분류 결과와 confidence를 본다
3. 필요시 카테고리를 수정한다
4. 청원을 published 또는 rejected 상태로 변경한다

## 6. 기능 요구사항

## 6.1 인증/계정

### 기능
- 이메일 기반 회원가입
- 로그인 / 로그아웃
- 역할 기반 접근 제어

### 역할
- guest
- user
- petition_creator
- moderator
- admin

### 완료 조건
- 인증 상태에 따라 접근 가능한 페이지가 다르게 동작해야 한다
- 관리자 라우트는 보호되어야 한다

---

## 6.2 청원 등록

### 입력 항목
- 제목
- 본문
- 대상기관
- 지역
- 해시태그
- 첨부자료(초기 mock 가능)

### 상태
- review
- published
- rejected
- closed

### 요구사항
- 등록 즉시 공개하지 않는다
- review 상태로 저장한다
- 자동분류 결과를 함께 저장한다
- confidence가 낮으면 review queue에 쌓인다

### 완료 조건
- 청원 등록 폼이 동작해야 한다
- 저장 후 목록/상세 조회 가능해야 한다
- 등록 시 자동분류 결과가 기록되어야 한다

---

## 6.3 청원 목록/상세

### 목록 필수 요소
- 제목
- 요약
- 상태
- 대표 카테고리
- 서명 수
- 생성일

### 필터
- 대분류
- 중분류
- 지역
- 상태
- 최신순 / 인기순

### 상세 필수 요소
- 제목
- 본문
- 카테고리
- 대상기관
- 서명 수
- 업데이트 섹션
- 후원 영역
- 상태

### 완료 조건
- 필터와 목록 조회가 정상 동작해야 한다
- 상세 페이지에서 서명과 후원 흐름으로 연결되어야 한다

---

## 6.4 서명 기능

### 수집 정보
- displayName 또는 name
- region
- ageBand
- gender
- consent flags

### 원칙
- 개인정보 최소 수집
- 연령은 생년월일 대신 연령대
- 성별은 응답거부 허용
- 중복 서명 방지 필요

### 완료 조건
- 한 사용자가 같은 청원에 중복 서명하는 기본 케이스를 방지해야 한다
- 서명 성공 시 서명 수가 반영되어야 한다
- 통계 집계 가능 구조로 저장되어야 한다

---

## 6.5 자동분류 MVP

### 목표
청원 제목과 본문을 바탕으로 대분류/중분류/소분류 후보를 자동 제안한다.

### 초기 전략
- 룰 기반
- 키워드 사전
- 동의어 사전
- 기관명/지역명 보정
- confidence 계산
- review queue 연계

### 출력
- 분류 후보 목록
- 대표 분류
- confidence
- review 필요 여부

### 완료 조건
- 등록 시 자동분류 함수가 호출되어야 한다
- 결과가 저장되어야 한다
- confidence 기준으로 review 여부가 정해져야 한다

---

## 6.6 검토 큐(Review Queue)

### 목적
자동분류 결과가 애매하거나 정책상 검토가 필요한 청원을 운영자가 확인하기 위한 큐

### 표시 정보
- 청원 제목
- 작성자
- 자동분류 결과
- confidence
- 상태
- 생성일

### 운영자 액션
- 카테고리 수정
- 승인
- 반려
- 상태 변경

### 완료 조건
- 운영자가 review 대상 청원을 목록으로 볼 수 있어야 한다
- 운영자가 분류 수정 후 publish 가능해야 한다

---

## 6.7 후원 기능(MVP)

### MVP 범위
- 도메인 모델 설계
- 청원별 후원 구조
- 플랫폼 후원 구조
- mock 결제 성공 흐름

### 후원 타입
- one-time
- recurring

### 완료 조건
- 후원 정보를 저장할 수 있어야 한다
- 청원별/플랫폼별 구분 가능해야 한다
- 향후 실제 PG adapter를 붙일 수 있는 구조여야 한다

---

## 6.8 운영자 대시보드

### 필수 지표
- 총 청원 수
- 활성 청원 수
- 상태별 청원 수
- 총 서명 수
- 지역별 서명 분포
- 연령대 분포
- 성별 분포
- 카테고리별 청원 수
- 총 후원액
- review queue 건수

### 원칙
- 개인정보 보호를 위해 너무 작은 표본은 기타 처리 또는 마스킹
- 운영자만 접근 가능

### 완료 조건
- 실제 DB 집계 기반으로 카드/차트가 렌더링되어야 한다
- 기간/카테고리 단위의 기본 필터 확장이 가능해야 한다

---

## 7. 비기능 요구사항

### 7.1 성능
- 목록 페이지는 페이지네이션 가능해야 한다
- 대시보드 집계는 분리된 서비스에서 계산한다
- lookup 데이터는 캐싱 가능 구조를 고려한다

### 7.2 보안
- 입력 검증 필수
- 관리자 라우트 보호
- 민감정보 최소 수집
- 파일 업로드 검증 구조 고려

### 7.3 유지보수성
- 프론트/백/온톨로지/공통 타입을 분리
- 타입 중복 최소화
- 도메인 로직은 UI에 넣지 않는다

### 7.4 접근성
- 모바일 우선
- 폼 라벨과 오류 메시지 명확화
- 키보드 접근성 고려

---

## 8. 성공 지표

MVP 단계의 성공 지표는 다음과 같다.

- 청원 생성 → 검토 → 공개 흐름이 동작한다
- 청원 상세에서 서명까지 자연스럽게 진행된다
- 자동분류가 운영자 검토를 보조할 수 있다
- 운영자가 대시보드를 통해 의미 있는 집계를 볼 수 있다
- 후원 구조가 추후 실결제로 확장 가능한 수준으로 설계된다

---

## 9. 수용 기준

다음 조건을 만족하면 MVP 완료로 본다.

1. 사용자가 청원을 등록할 수 있다
2. 청원이 review 상태로 저장된다
3. 시스템이 자동분류 결과를 생성한다
4. 운영자가 review queue에서 청원을 승인/반려할 수 있다
5. 공개된 청원을 사용자가 조회할 수 있다
6. 공개된 청원에 사용자가 서명할 수 있다
7. 운영자가 서명 통계를 대시보드에서 볼 수 있다
8. 후원 도메인과 mock 흐름이 동작한다

## 12. 최종 정의

이 MVP는 “완성된 대형 청원 플랫폼”이 아니라,  
**청원 + 자동분류 + 서명 + 통계 + 후원 확장 구조**를 실제로 검증할 수 있는 최소 기능 제품이다.
