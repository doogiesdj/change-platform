# Permission Matrix
Ontology 기반 청원 + 후원 플랫폼 인증/권한 매트릭스

## 1. 문서 목적

이 문서는 플랫폼의 사용자 역할(Role)과 권한(Permission)을 정의한다.  
목표는 다음과 같다.

- 인증/인가 기준을 일관되게 유지한다
- 프론트엔드 UI 노출 규칙과 백엔드 접근 제어 기준을 맞춘다
- 관리자/운영자 기능을 일반 사용자 기능과 명확히 분리한다
- 구현 시 Guard, Middleware, UI 분기, 메뉴 노출 정책의 기준으로 사용한다

이 문서는 MVP 기준 문서이며, 실제 구현 시 Auth Module, Route Guard, Admin UI 정책과 함께 동기화되어야 한다.

---

## 2. 기본 원칙

### 2.1 최소 권한 원칙
사용자는 자신의 역할에 필요한 최소 기능만 사용할 수 있어야 한다.

### 2.2 공개/운영 분리
공개 페이지와 운영 페이지는 명확히 분리한다.

### 2.3 서버 검증 우선
프론트엔드에서 버튼을 숨기더라도, 실제 권한 검증은 반드시 백엔드에서 수행한다.

### 2.4 역할보다 정책 우선
역할만으로 설명이 부족한 경우, 리소스 소유권(본인 청원 여부)과 상태값(공개 여부)을 함께 고려한다.

---

## 3. 역할 정의

## 3.1 guest
비로그인 사용자

### 설명
- 공개된 청원 목록/상세를 조회할 수 있는 사용자
- 인증이 필요한 기능은 사용할 수 없음

---

## 3.2 user
기본 로그인 사용자

### 설명
- 청원 서명
- 후원
- 개인 활동 확인
- 기본 사용자 기능 수행 가능

---

## 3.3 petition_creator
청원 작성 권한을 가진 사용자

### 설명
- user의 권한을 모두 포함
- 청원 생성 및 자신의 청원 관리 가능

> 참고: MVP에서는 `user`와 `petition_creator`를 통합할 수도 있다.  
> 단, 문서상 분리는 유지해두는 것이 확장에 유리하다.

---

## 3.4 moderator
운영 검토 담당자

### 설명
- review queue 접근
- 자동분류 결과 검토
- 청원 승인/반려/재분류 가능
- 운영 도구 사용 가능

---

## 3.5 admin
최고 관리자

### 설명
- 모든 운영자 기능 포함
- 전체 대시보드
- 관리자 로그 조회
- 운영 정책 설정
- 권한 관리 확장 가능

---

## 4. 리소스 소유권 개념

일부 권한은 단순 역할이 아니라 **소유권**에 따라 달라진다.

예:
- 본인이 작성한 청원은 수정 가능
- 다른 사용자의 청원은 수정 불가
- 운영자는 다른 사용자의 청원도 검토 가능

### ownership 조건 예시
- `isOwner`: 현재 로그인 사용자가 해당 청원의 작성자인가
- `isModeratorOrAdmin`: moderator 또는 admin인가
- `isPublished`: 청원이 공개 상태인가
- `isReviewable`: review 상태인가

---

## 5. 주요 상태값과 권한 관계

## 5.1 Petition.status
- `review`
- `published`
- `rejected`
- `closed`
- `achieved`

### 해석
- `review`: 운영 검토 전, 일반 공개 금지
- `published`: 공개 가능
- `rejected`: 반려, 공개 금지
- `closed`: 종료, 서명 제한 가능
- `achieved`: 성과 달성 상태, 정책에 따라 공개 유지 가능

## 7. API 권한 기준

## 7.1 공개 API
인증 없이 사용 가능한 API

예:
- `GET /api/petitions`
- `GET /api/petitions/:id`
- `GET /api/categories/tree`
- `GET /api/regions`

단, 청원 목록/상세는 기본적으로 `published` 상태만 반환

---

## 7.2 인증 필요 API
로그인 필요

예:
- `POST /api/petitions`
- `POST /api/petitions/:id/signatures`
- `POST /api/petitions/:id/donations`
- `GET /api/auth/me`

---

## 7.3 운영 API
moderator 이상 필요

예:
- `GET /api/admin/review-queue`
- `PATCH /api/admin/review-queue/:id/approve`
- `PATCH /api/admin/review-queue/:id/reject`
- `POST /api/admin/petitions/:id/reclassify`

---

## 7.4 관리자 API
admin 전용

예:
- `GET /api/admin/audit-logs`
- 운영 설정 변경 API
- 역할 변경 API

## 9. 리소스 소유권 기반 예외 규칙

다음 기능은 역할만으로 판단하지 않고 **본인 소유 여부**를 함께 본다.

### 9.1 청원 수정
- 작성자 본인 가능
- moderator는 상태/분류 관련 운영 수정만 허용 권장
- admin은 전체 가능

### 9.2 청원 업데이트 작성
- 작성자 본인 가능
- admin 가능
- moderator는 정책상 필요 시만 허용

### 9.3 rejected/review 청원 조회
- 작성자 본인 가능
- moderator / admin 가능
- 일반 공개 불가

## 10. 개인정보 보호 관점 권한 규칙

### 10.1 서명 데이터
- 개별 서명 원시 데이터는 공개 금지
- 운영자도 필요 최소 범위만 접근
- 집계 화면은 소표본 마스킹 필요

### 10.2 후원 데이터
- 개별 후원 상세 정보는 공개 금지
- 운영자/관리자도 민감정보 최소 접근 원칙 적용
- 공개 화면은 총액/집계 중심

### 10.3 감사 로그
- 관리자 전용
- 개인정보가 포함된 before/after는 마스킹 고려

## 13. MVP 권장 단순화 정책

초기 MVP에서는 다음처럼 단순화할 수 있다.

### 단순화 1
`user`와 `petition_creator`를 동일 역할로 취급

### 단순화 2
서명/후원은 로그인 사용자만 허용

### 단순화 3
운영 권한은 `moderator`, `admin` 두 단계만 유지

### 단순화 4
카테고리 마스터 수정은 admin만 가능

이 단순화는 구현 복잡도를 낮추는 데 도움이 된다.

---

## 14. 테스트 체크리스트

권한 관련 테스트 시 최소 확인 항목:

- [ ] guest는 review 청원에 접근할 수 없는가
- [ ] user는 본인 청원만 수정 가능한가
- [ ] moderator는 review queue에 접근 가능한가
- [ ] admin만 audit log에 접근 가능한가
- [ ] 서명/후원 기능은 로그인 사용자만 가능한가
- [ ] 공개되지 않은 청원은 일반 사용자에게 숨겨지는가
- [ ] 역할은 맞지만 소유권이 없는 경우 제한되는가

## 15. 결론

이 문서의 핵심은 다음과 같다.

1. 공개 영역과 운영 영역을 분리한다
2. 역할 기반 권한과 소유권 기반 권한을 함께 고려한다
3. 개인정보 보호를 권한 설계에 포함한다
4. 프론트 노출과 백엔드 검증을 함께 설계한다

즉, 이 권한 매트릭스는 단순 메뉴 표시 문서가 아니라  
**서비스의 보안, 운영, 개인정보 보호 정책을 동시에 반영하는 기준 문서**다.
